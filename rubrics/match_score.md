# JD-简历匹配度评分规则

## 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 硬性条件匹配 | 20% | 学历、年限、证书等 |
| 技能匹配 | 25% | 技术技能、工具熟练度 |
| 经验相关度 | 30% | 工作经历与岗位职责的相关度 |
| 成果量化程度 | 15% | 数据化成果展示 |
| 表达清晰度 | 10% | 经历描述的专业程度 |

## 评分标准

### 1. 硬性条件匹配（20 分）

| 分值 | 标准 |
|------|------|
| 20 | 所有硬性条件完全满足 |
| 15 | 满足 80%+ 硬性条件 |
| 10 | 满足 50-80% 硬性条件 |
| 5 | 满足 50% 以下硬性条件 |
| 0 | 完全不满足硬性条件 |

**评估要点**：
- 学历要求是否满足
- 工作年限要求是否满足
- 证书、语言等硬性要求是否满足
- 标注"必须"的条件是否满足

**风险提示**：硬性条件不满足时，必须标注"可能被机筛淘汰"

---

### 2. 技能匹配（25 分）

| 分值 | 标准 |
|------|------|
| 25 | 核心技能全部覆盖，且有深度 |
| 20 | 核心技能 80%+ 覆盖 |
| 15 | 核心技能 50-80% 覆盖 |
| 10 | 核心技能 50% 以下覆盖 |
| 5 | 仅基础技能覆盖 |
| 0 | 核心技能完全未覆盖 |

**评估要点**：
- 技术栈匹配度（SQL、Python、Java 等）
- 工具熟练度（Axure、Figma、Tableau 等）
- 技能深度而非仅列出名称
- 技能必须有项目证据

**证据要求**：
- 技能出现在技能区但无项目证据 → 最多 15 分
- 技能有项目证据但描述模糊 → 最多 20 分
- 技能有完整项目证据 → 满分

---

### 3. 经验相关度（30 分）

| 分值 | 标准 |
|------|------|
| 30 | 主要经历与岗位职责高度匹配 |
| 25 | 主要经历 80%+ 相关 |
| 20 | 主要经历 50-80% 相关 |
| 15 | 主要经历 50% 以下相关 |
| 10 | 仅边缘经历相关 |
| 0 | 经历与岗位完全无关 |

**评估要点**：
- 工作经历行业相关性
- 项目类型匹配度
- 职责范围重叠度
- 可迁移能力的正确表述

**证据要求**：
- 直接经验 > 可迁移经验 > 学习/作品证据
- 不得将可迁移经验等同于直接经验

---

### 4. 成果量化程度（15 分）

| 分值 | 标准 |
|------|------|
| 15 | 所有核心经历有量化成果 |
| 12 | 80%+ 经历有量化成果 |
| 10 | 50-80% 经历有量化成果 |
| 7 | 50% 以下经历有量化成果 |
| 4 | 有成果但无量化 |
| 0 | 无成果展示 |

**评估要点**：
- 数据指标的具体性
- 成果的可验证性
- 归因的合理性
- 估算数据的标注

**证据要求**：
- 有数据无来源 → 最多 10 分
- 有数据有来源 → 满分
- 定性成果（已上线、被采纳）→ 最多 7 分

---

### 5. 表达清晰度（10 分）

| 分值 | 标准 |
|------|------|
| 10 | 表达专业、结构清晰、重点突出 |
| 8 | 表达较好，有少量改进空间 |
| 6 | 表达一般，有明显问题 |
| 4 | 表达混乱，结构不清 |
| 0 | 完全无法理解 |

**评估要点**：
- 职责化 vs 成果化表达
- 贡献边界的清晰度
- 关键词的自然嵌入
- 无 AI 模板腔

---

## 综合评分

### 计算公式

```
综合匹配分 = 硬性条件×0.20 + 技能匹配×0.25 + 经验相关度×0.30 + 成果量化×0.15 + 表达清晰度×0.10
```

### 分级标准

| 分数区间 | 级别 | 说明 |
|----------|------|------|
| 90-100 | 极强匹配 | 强烈推荐投递 |
| 80-89 | 强匹配 | 推荐投递 |
| 70-79 | 中等匹配 | 可以投递，需准备补充材料 |
| 60-69 | 弱匹配 | 谨慎投递，需要针对性准备 |
| 50-59 | 勉强匹配 | 不建议作为首选 |
| < 50 | 不匹配 | 建议寻找更匹配的岗位 |

---

## 评分约束

### 必须遵守

1. **必须有 JD 原文依据**：每项评分必须引用 JD 原文
2. **必须有简历证据**：每项评分必须引用简历具体内容
3. **不猜分**：信息不足时标注"暂无法评分"
4. **不默认满分**：缺失信息必须扣分
5. **不平均**：无法评分的维度不能给 60% 默认分

### 禁止行为

- 根据岗位名称预设匹配度
- 将弱匹配写为强匹配
- 忽略硬性条件的缺失
- 给无法验证的技能打满分

## 评分输出格式

```markdown
## JD 匹配评分

**综合匹配分：78/100**

| 维度 | 得分 | 满分 | 证据 |
|------|------|------|------|
| 硬性条件 | 18 | 20 | JD要求3年+，简历显示4年+ |
| 技能匹配 | 18 | 25 | SQL弱覆盖，PRD/原型强覆盖 |
| 经验相关度 | 25 | 30 | B端订单经验直接匹配 |
| 成果量化 | 10 | 15 | 60%经历有量化数据 |
| 表达清晰度 | 7 | 10 | 整体较好，偶有职责化表述 |

### 关键扣分原因

1. SQL 深度不足（-5）
2. Python 无证据（-5）
3. 部分经历缺数据支撑（-5）
```

## 风险提示规则

### 必须提示的风险

当以下情况发生时，必须在报告中明确提示：

1. **硬性条件缺失**：可能导致机筛淘汰
2. **核心技能缺失**：面试阶段可能被质疑
3. **数据来源不明**：可能导致信任度下降
4. **夸大表述**：可能导致面试失败
5. **转行硬凑**：可能导致面试不通过

### 提示格式

```markdown
### ⚠️ 风险提示

**高风险（影响初筛）**：
- 缺少 Python 经验，机筛可能淘汰

**中风险（影响面试）**：
- SQL 深度描述不足，可能被追问

**低风险（可准备）**：
- A/B 测试经验，可补充学习项目
```
